# Add Clone-Based Container Isolation to Gator

**Date**: 2026-02-19
**Status**: Proposed

## Problem

Gator's current `container` isolation mode uses copy-in/copy-out (tar + `docker cp`) to move files between a host worktree and a sandboxed Docker container. This works but has practical limitations at scale:

1. **Slow for large repos**: The entire working tree is tarred in before the agent starts and tarred out after it finishes. For a repo with 10,000 files this adds minutes of overhead per task.

2. **No git inside the container**: `.git` is excluded from the copy, so the agent cannot run git operations (commit, diff, log, blame). Claude Code uses git operations internally for context gathering, so this degrades agent performance.

3. **No real-time visibility**: The host cannot observe the agent's file changes until extraction. This prevents live monitoring and makes debugging harder.

4. **No resource limits**: Containers run with unlimited memory and CPU. With 10+ concurrent agents, a single runaway agent can starve the others.

5. **Fixed container images**: `container_image` is a string pointing to a pre-built image. There is no mechanism for projects to extend the base image with project-specific toolchains (Rust, Go, etc.) and have gator build/cache the result.

## Background Research

### Why Not Worktrees in Containers?

Git worktrees store `gitdir: /absolute/host/path` in their `.git` file. When a worktree directory is bind-mounted into a container at `/workspace`, this absolute path is unresolvable inside the container and all git operations fail. This is why the current container mode excludes `.git` and uses copy-in/copy-out instead.

### Standalone Clones

`git clone --dissociate --reference <source> <source> <dest>` creates a fully self-contained git repository that:
- Borrows objects from the source repo during clone (fast)
- Detaches from the source after clone (`--dissociate`) so no runtime dependency
- Has its own `.git` directory with no absolute host path references
- Works perfectly with container bind mounts

The clone is fast because `--reference` avoids copying objects that already exist in the source. `--dissociate` then copies the borrowed objects so the clone is fully standalone. For a 1 GB repo, the clone typically completes in 1-2 seconds on NVMe.

### Container Runtimes on macOS

Tested with OrbStack (already installed on the development machine). Key properties:
- Docker-compatible API (standard `docker` CLI commands work)
- Millisecond container starts (vs 20-30s for Docker Desktop initialization)
- Near-native filesystem performance via VirtioFS for bind mounts
- Under 1 GB base RAM for the VM

The implementation uses standard Docker CLI commands, so Colima, Docker Desktop, or Podman work as drop-in alternatives.

### Security Model

The new mode sits between the existing modes on the isolation spectrum:

| Mode | Host FS Access | Process Isolation | Git Inside | Copy Overhead |
|------|---------------|-------------------|------------|---------------|
| `worktree` | Full (host worktree) | None | Yes | None |
| `clone` | Scoped (clone dir only) | Container | Yes | None |
| `container` | None | Container | No | Full copy in+out |

The `clone` mode's bind mount exposes only the standalone clone directory to the container. The agent cannot reach the source repository, host home directory, or any other host path through this mount. Process and (eventually) network isolation are provided by the container boundary.

## Design

### New Isolation Mode: `"clone"`

A third value for the `isolation` field in plan TOML. The existing `"worktree"` and `"container"` modes are unchanged.

```toml
[plan]
name = "add-authentication"
base_branch = "main"
isolation = "clone"
container_image = "gator-agent:latest"
container_memory_mb = 4096
container_cpus = 1.5
```

### Workspace Lifecycle

#### Create

1. Determine clone directory: `<repo-name>-gator-clones/<plan>-<task>/`
   (follows existing convention: worktrees go in `<repo-name>-gator-worktrees/`)

2. Create standalone clone:
   ```
   git clone --dissociate --reference <repo_path> \
     --single-branch --branch <base_branch> \
     <repo_path> <clone_dir>
   ```

3. Create agent branch:
   ```
   git -C <clone_dir> checkout -b gator/<plan>/<task>
   ```

4. Create and start container with bind mount:
   ```
   docker create \
     --name gator-<plan>-<task> \
     -v <clone_dir>:/workspace \
     -w /workspace \
     --memory <limit>m \
     --cpus <limit> \
     <image> sleep infinity
   docker start <container_id>
   ```

5. Configure git inside container:
   ```
   docker exec <cid> git config --global --add safe.directory /workspace
   ```

6. Return `WorkspaceInfo`:
   - `path`: `/workspace` (what the agent sees)
   - `host_path`: `<clone_dir>` (for gate checks on host)
   - `branch`: `gator/<plan>/<task>`
   - `container_id`: from docker create

#### Extract Results

After the agent finishes, push the branch back to the source repository so that `gator merge` can find it:

1. Commit any uncommitted changes in the clone (the agent may or may not have committed its own work):
   ```
   git -C <clone_dir> add -A
   git -C <clone_dir> status --porcelain  # check if anything to commit
   git -C <clone_dir> commit -m "gator: <task> (attempt <n>)"
   ```

2. Check for commits on the agent branch:
   ```
   git -C <clone_dir> rev-list --count <base_branch>..gator/<plan>/<task>
   ```

3. If commits exist, push branch to source repo:
   ```
   git -C <clone_dir> push <repo_path> gator/<plan>/<task>:gator/<plan>/<task> --force
   ```
   `--force` is safe because gator owns these branch names and each session gets a unique branch.

4. If no commits exist, mark the task's workspace as empty (no changes to merge).

After this step, the branch exists in the source repository and the rest of gator's workflow (`gator merge`, `gator pr`) works unchanged.

#### Remove

1. Stop and remove container: `docker rm -f <container_id>`
2. Remove clone directory: `rm -rf <clone_dir>`

### Resource Limits

New struct in `crates/gator-core/src/isolation/mod.rs`:

```rust
#[derive(Debug, Clone, Default)]
pub struct ResourceLimits {
    /// Container memory limit in megabytes (maps to docker --memory).
    pub memory_mb: Option<u32>,
    /// Container CPU limit (maps to docker --cpus). 1.5 means 1.5 cores.
    pub cpus: Option<f32>,
}
```

Added to `ContainerConfig` (used by both `ContainerIsolation` and `CloneIsolation`):

```rust
pub struct ContainerConfig {
    pub image: String,
    pub extra_flags: Vec<String>,
    pub resource_limits: ResourceLimits,  // NEW
}
```

When building `docker create` arguments, if `memory_mb` is set, add `--memory <n>m`. If `cpus` is set, add `--cpus <n>`.

Suggested defaults for a machine with 128 GB RAM / 16 cores:

| Concurrent Tasks | Memory per Container | CPUs per Container |
|-----------------|--------------------|--------------------|
| 10              | 4096 MB            | 1.5                |
| 15              | 2048 MB            | 1.0                |
| 20              | 2048 MB            | 0.75               |

No auto-calculation â€” the operator sets limits explicitly in the plan TOML. Gator logs a warning at dispatch time if the total committed resources exceed 80% of host capacity. Host capacity is read via `sysctl hw.memsize` (macOS) or `/proc/meminfo` (Linux).

### Plan TOML Changes

New optional fields in `[plan]`:

```toml
[plan]
name = "my-plan"
base_branch = "main"
isolation = "clone"                 # NEW valid value
container_image = "gator-agent:latest"
container_memory_mb = 4096          # NEW (optional)
container_cpus = 1.5                # NEW (optional)
```

In `PlanMeta` (`crates/gator-core/src/plan/toml_format.rs`):

```rust
#[serde(default, skip_serializing_if = "Option::is_none")]
pub container_memory_mb: Option<u32>,

#[serde(default, skip_serializing_if = "Option::is_none")]
pub container_cpus: Option<f32>,
```

Validation in `parser.rs`: accept `"clone"` alongside `"worktree"` and `"container"` for the `isolation` field. Warn if `container_memory_mb` or `container_cpus` are set but `isolation` is `"worktree"` (they have no effect).

### Database Migration

`crates/gator-db/migrations/006_clone_isolation.sql`:

```sql
-- Resource limits for container and clone isolation modes.
ALTER TABLE plans ADD COLUMN container_memory_mb INTEGER;
ALTER TABLE plans ADD COLUMN container_cpus REAL;
```

Both columns are nullable (NULL means no limit). Update `models.rs` Plan struct and `queries/plans.rs` insert/select to include these columns.

### Factory Function Changes

`crates/gator-core/src/isolation/mod.rs`:

The `create_isolation` function signature needs to accept resource limits:

```rust
pub fn create_isolation(
    mode: &str,
    repo_path: &Path,
    container_image: Option<&str>,
    resource_limits: ResourceLimits,  // NEW parameter
) -> Result<Arc<dyn Isolation>>
```

New match arm:

```rust
"clone" => {
    let image = container_image.unwrap_or("ubuntu:24.04").to_string();
    let config = ContainerConfig {
        image,
        extra_flags: vec![],
        resource_limits,
    };
    Ok(Arc::new(clone::CloneIsolation::new(config, repo_path, None)?))
}
```

The existing `"container"` arm also receives `resource_limits` so the current mode gains resource limit support too.

### Lifecycle Ordering Change

The current lifecycle (in `crates/gator-core/src/lifecycle/mod.rs`) runs:

```
... â†’ agent finishes â†’ extract_results â†’ gate checks â†’ commit (if passed) â†’ done
```

For clone isolation, the agent may have committed directly (it has git access). The extract step needs to:
1. First commit any uncommitted work
2. Then push the branch

So `extract_results` for `CloneIsolation` internally calls the commit logic before pushing. The lifecycle code does not need to change â€” the `Isolation` trait's `extract_results` method encapsulates the difference.

The existing `commit_agent_work` call in the lifecycle (after gate pass) becomes a no-op for clone isolation because `extract_results` already committed and pushed. It already handles the "nothing to commit" case by returning `Ok(false)`.

### Project-Specific Dockerfile (Deferred)

Building project-specific images from a `.gator/Dockerfile` is useful but can be implemented as a follow-up. For now, operators build their images separately and reference them via `container_image`. This keeps the initial scope focused on the isolation mechanism.

When implemented later, the flow would be:
1. Check for `<project>/.gator/Dockerfile`
2. Hash its contents for a cache key: `gator-project-<hash>:latest`
3. If image doesn't exist, build it: `docker build -t gator-project-<hash>:latest -f .gator/Dockerfile .`
4. Use the built image for the plan

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Bind mount performance on Docker Desktop (non-OrbStack) | Document OrbStack as recommended runtime; VirtioFS is available on Docker Desktop 4.6+ as well |
| Clone disk usage with many concurrent tasks | `--dissociate --reference` keeps clones small; `gator cleanup` removes them |
| `git push --force` to source repo | Only pushes to `gator/<plan>/<task>` branches which are owned by gator; source repo's main branch is never force-pushed |
| Race condition: two tasks push to same branch | Branch names include plan name + task name, which are unique within a plan |
| Agent modifies `.git` inside container | Low risk â€” agent would have to intentionally corrupt git; the clone is disposable anyway |
